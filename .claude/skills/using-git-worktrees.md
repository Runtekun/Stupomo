# Git ワークツリーの使用

## 概要

作業が分離されたワークスペースで行われるようにする。プラットフォームのネイティブワークツリーツールを優先する。ネイティブツールが利用できない場合のみ、手動のgitワークツリーにフォールバックする。

**コア原則：** まず既存の分離を検出する。次にネイティブツールを使う。最後にgitにフォールバックする。ハーネスと戦わない。

**開始時に宣言する：** 「using-git-worktrees スキルを使用して分離されたワークスペースをセットアップします。」

## ステップ0：既存の分離を検出する

**何も作成する前に、すでに分離されたワークスペースにいるか確認する。**

```bash
GIT_DIR=$(cd "$(git rev-parse --git-dir)" 2>/dev/null && pwd -P)
GIT_COMMON=$(cd "$(git rev-parse --git-common-dir)" 2>/dev/null && pwd -P)
BRANCH=$(git branch --show-current)
```

**サブモジュールのガード：** `GIT_DIR != GIT_COMMON` はgitサブモジュールの内部でも真になる。「すでにワークツリーにいる」と結論付ける前に、サブモジュールでないことを確認する：

```bash
# これがパスを返した場合、ワークツリーではなくサブモジュールにいる — 通常のリポジトリとして扱う
git rev-parse --show-superproject-working-tree 2>/dev/null
```

**`GIT_DIR != GIT_COMMON`（サブモジュールでない場合）：** すでにリンクされたワークツリーにいる。ステップ3（プロジェクトセットアップ）へスキップする。別のワークツリーを作成しない。

ブランチ状態とともに報告する：
- ブランチ上：「`<パス>` のブランチ `<名前>` ですでに分離されたワークスペースにいます。」
- デタッチHEAD：「`<パス>` ですでに分離されたワークスペースにいます（デタッチHEAD、外部管理）。終了時にブランチの作成が必要です。」

**`GIT_DIR == GIT_COMMON`（またはサブモジュール内）：** 通常のリポジトリチェックアウトにいる。

ユーザーが指示でワークツリーの好みをすでに示しているか？ そうでない場合、作成前に同意を求める：

> 「分離されたワークツリーをセットアップしましょうか？ 現在のブランチへの変更を保護します。」

宣言された既存の好みは確認せずに守る。ユーザーが同意しない場合、その場で作業してステップ3へスキップする。

## ステップ1：分離されたワークスペースを作成する

**2つのメカニズムがある。この順番で試みる。**

### 1a. ネイティブワークツリーツール（推奨）

ユーザーが分離されたワークスペースを求めた（ステップ0の同意）。ワークツリーを作成する方法がすでにあるか？ `EnterWorktree`・`WorktreeCreate`・`/worktree` コマンド・`--worktree` フラグのような名前のツールかもしれない。ある場合はそれを使ってステップ3へスキップする。

ネイティブツールはディレクトリの配置・ブランチの作成・クリーンアップを自動的に処理する。ネイティブツールがある場合に `git worktree add` を使うと、ハーネスが見えない・管理できないファントム状態が作られる。

ネイティブワークツリーツールが利用できない場合のみステップ1bへ進む。

### 1b. Git ワークツリーのフォールバック

**ステップ1aが適用できない場合のみ使用する** — ネイティブワークツリーツールが利用できない。gitを使って手動でワークツリーを作成する。

#### ディレクトリの選択

この優先順位に従う。明示的なユーザーの好みは常に観察されたファイルシステムの状態より優先する。

1. **宣言されたワークツリーディレクトリの好みを指示で確認する。** ユーザーがすでに指定している場合、確認なしにそれを使用する。

2. **既存のプロジェクトローカルワークツリーディレクトリを確認する：**
   ```bash
   ls -d .worktrees 2>/dev/null     # 推奨（隠し）
   ls -d worktrees 2>/dev/null      # 代替
   ```
   見つかった場合はそれを使用する。両方ある場合は `.worktrees` が優先。

3. **既存のグローバルディレクトリを確認する：**
   ```bash
   project=$(basename "$(git rev-parse --show-toplevel)")
   ls -d ~/.config/superpowers/worktrees/$project 2>/dev/null
   ```
   見つかった場合はそれを使用する（レガシーグローバルパスとの後方互換性）。

4. **他のガイダンスがない場合**、プロジェクトルートの `.worktrees/` をデフォルトとする。

#### 安全確認（プロジェクトローカルディレクトリのみ）

**ワークツリーを作成する前にディレクトリが無視されていることを確認しなければならない：**

```bash
git check-ignore -q .worktrees 2>/dev/null || git check-ignore -q worktrees 2>/dev/null
```

**無視されていない場合：** .gitignore に追加し、変更をコミットしてから進む。

**なぜ重要か：** ワークツリーの内容を誤ってリポジトリにコミットすることを防ぐ。

グローバルディレクトリ（`~/.config/superpowers/worktrees/`）は確認不要。

#### ワークツリーの作成

```bash
project=$(basename "$(git rev-parse --show-toplevel)")

# 選択した場所に基づいてパスを決定
# プロジェクトローカルの場合：path="$LOCATION/$BRANCH_NAME"
# グローバルの場合：path="~/.config/superpowers/worktrees/$project/$BRANCH_NAME"

git worktree add "$path" -b "$BRANCH_NAME"
cd "$path"
```

**サンドボックスのフォールバック：** `git worktree add` が権限エラーで失敗した場合（サンドボックスの拒否）、サンドボックスがワークツリーの作成をブロックしたことをユーザーに伝え、代わりに現在のディレクトリで作業する。その後セットアップとベースラインテストをその場で実行する。

## ステップ3：プロジェクトセットアップ

自動検出して適切なセットアップを実行する：

```bash
# Node.js
if [ -f package.json ]; then npm install; fi

# Rust
if [ -f Cargo.toml ]; then cargo build; fi

# Python
if [ -f requirements.txt ]; then pip install -r requirements.txt; fi
if [ -f pyproject.toml ]; then poetry install; fi

# Go
if [ -f go.mod ]; then go mod download; fi
```

## ステップ4：クリーンなベースラインを確認する

ワークスペースがクリーンな状態で開始されることを確認するためにテストを実行する：

```bash
# プロジェクトに適したコマンドを使用する
npm test / cargo test / pytest / go test ./...
```

**テストが失敗した場合：** 失敗を報告し、続けるか調査するかを尋ねる。

**テストが通過した場合：** 準備完了を報告する。

### レポート

```
<フルパス> でワークツリーが準備完了
テスト通過（N件のテスト、0件の失敗）
<機能名> を実装する準備ができました
```

## クイックリファレンス

| 状況 | アクション |
|------|----------|
| すでにリンクされたワークツリーにいる | 作成をスキップ（ステップ0） |
| サブモジュール内にいる | 通常のリポジトリとして扱う（ステップ0のガード） |
| ネイティブワークツリーツールが利用可能 | それを使用（ステップ1a） |
| ネイティブツールなし | gitワークツリーのフォールバック（ステップ1b） |
| `.worktrees/` が存在する | それを使用（無視されていることを確認） |
| `worktrees/` が存在する | それを使用（無視されていることを確認） |
| 両方が存在する | `.worktrees/` を使用 |
| どちらも存在しない | 指示ファイルを確認、次に `.worktrees/` をデフォルト |
| グローバルパスが存在する | それを使用（後方互換性） |
| ディレクトリが無視されていない | .gitignore に追加してコミット |
| 作成で権限エラー | サンドボックスのフォールバック、その場で作業 |
| ベースライン中にテストが失敗 | 失敗を報告して確認する |
| package.json/Cargo.toml なし | 依存関係のインストールをスキップ |

## よくある間違い

### ハーネスと戦う

- **問題：** プラットフォームがすでに分離を提供しているのに `git worktree add` を使用する
- **修正：** ステップ0が既存の分離を検出する。ステップ1aがネイティブツールに委譲する。

### 検出のスキップ

- **問題：** 既存のワークツリー内にネストされたワークツリーを作成する
- **修正：** 何かを作成する前に必ずステップ0を実行する

### 無視確認のスキップ

- **問題：** ワークツリーの内容が追跡され、gitの状態を汚染する
- **修正：** プロジェクトローカルワークツリーを作成する前に必ず `git check-ignore` を使用する

### ディレクトリの場所を仮定する

- **問題：** 不一貫性が生まれ、プロジェクトの規約に違反する
- **修正：** 優先順位に従う：既存 > グローバルレガシー > 指示ファイル > デフォルト

### 失敗するテストで進む

- **問題：** 新しいバグと既存の問題を区別できない
- **修正：** 失敗を報告して、続けるかの明示的な許可を得る

## 禁止事項

**絶対にしないこと：**
- ステップ0が既存の分離を検出したのにワークツリーを作成する
- ネイティブワークツリーツール（例：`EnterWorktree`）がある場合に `git worktree add` を使用する。これが最大の間違いだ — あるなら使う。
- ステップ1aをスキップしてステップ1bのgitコマンドに直接ジャンプする
- 無視されていることを確認せずにワークツリーを作成する（プロジェクトローカル）
- ベースラインのテスト確認をスキップする
- 確認なしに失敗するテストで進む

**必ずすること：**
- 最初にステップ0の検出を実行する
- gitのフォールバックよりネイティブツールを優先する
- ディレクトリの優先順位に従う：既存 > グローバルレガシー > 指示ファイル > デフォルト
- プロジェクトローカルのディレクトリが無視されていることを確認する
- プロジェクトセットアップを自動検出して実行する
- クリーンなテストベースラインを確認する

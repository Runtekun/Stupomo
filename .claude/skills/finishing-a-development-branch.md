# 開発ブランチの完了

## 概要

明確な選択肢を提示して処理することで、開発作業の完了をガイドする。

**コア原則：** テストを確認する → 環境を検出する → 選択肢を提示する → 選択を実行する → クリーンアップ。

**開始時に宣言する：** 「finishing-a-development-branch スキルを使用してこの作業を完了させます。」

## プロセス

### ステップ1：テストを確認する

**選択肢を提示する前に、テストが通過することを確認する：**

```bash
# プロジェクトのテストスイートを実行する
npm test / cargo test / pytest / go test ./...
```

**テストが失敗した場合：**
```
テスト失敗中（N件の失敗）。完了前に修正が必要です：

[失敗を表示]

テストが通過するまでマージ/PRを進めることはできません。
```

停止する。ステップ2へ進まない。

**テストが通過した場合：** ステップ2へ続く。

### ステップ2：環境を検出する

**選択肢を提示する前にワークスペースの状態を確認する：**

```bash
GIT_DIR=$(cd "$(git rev-parse --git-dir)" 2>/dev/null && pwd -P)
GIT_COMMON=$(cd "$(git rev-parse --git-common-dir)" 2>/dev/null && pwd -P)
```

これにより表示するメニューとクリーンアップの方法が決まる：

| 状態 | メニュー | クリーンアップ |
|------|----------|--------------|
| `GIT_DIR == GIT_COMMON`（通常リポジトリ） | 標準4選択肢 | ワークツリーなし |
| `GIT_DIR != GIT_COMMON`、名前付きブランチ | 標準4選択肢 | 出所ベース（ステップ6参照） |
| `GIT_DIR != GIT_COMMON`、デタッチHEAD | 縮小3選択肢（マージなし） | クリーンアップなし（外部管理） |

### ステップ3：ベースブランチを確認する

```bash
# 一般的なベースブランチを試す
git merge-base HEAD main 2>/dev/null || git merge-base HEAD master 2>/dev/null
```

または確認する：「このブランチは main から分岐しています。合っていますか？」

### ステップ4：選択肢を提示する

**通常リポジトリおよび名前付きブランチのワークツリー — 正確にこの4つの選択肢を提示する：**

```
実装完了。どうしますか？

1. <ベースブランチ> にローカルでマージする
2. プッシュしてプルリクエストを作成する
3. ブランチをそのままにする（後で対応します）
4. この作業を破棄する

どの選択肢にしますか？
```

**デタッチHEAD — 正確にこの3つの選択肢を提示する：**

```
実装完了。デタッチHEAD（外部管理のワークスペース）です。

1. 新しいブランチとしてプッシュしてプルリクエストを作成する
2. そのままにする（後で対応します）
3. この作業を破棄する

どの選択肢にしますか？
```

**説明は追加しない** — 選択肢は簡潔に。

### ステップ5：選択を実行する

#### 選択肢1：ローカルでマージする

```bash
# 安全のためにメインリポジトリのルートを取得
MAIN_ROOT=$(git -C "$(git rev-parse --git-common-dir)/.." rev-parse --show-toplevel)
cd "$MAIN_ROOT"

# まずマージ — 何かを削除する前に成功を確認する
git checkout <ベースブランチ>
git pull
git merge <フィーチャーブランチ>

# マージ結果でテストを確認する
<テストコマンド>

# マージ成功後のみ：ワークツリーをクリーンアップ（ステップ6）、その後ブランチを削除
```

その後：ワークツリーをクリーンアップ（ステップ6）、そしてブランチを削除：

```bash
git branch -d <フィーチャーブランチ>
```

#### 選択肢2：プッシュしてPRを作成する

```bash
# ブランチをプッシュ
git push -u origin <フィーチャーブランチ>

# PRを作成
gh pr create --title "<タイトル>" --body "$(cat <<'EOF'
## サマリー
<変更内容を2〜3項目で>

## テスト計画
- [ ] <確認手順>
EOF
)"
```

**ワークツリーをクリーンアップしない** — PRフィードバックを反映するためにユーザーが必要とする。

#### 選択肢3：そのままにする

報告する：「ブランチ <名前> を保持します。ワークツリーは <パス> に保存されています。」

**ワークツリーをクリーンアップしない。**

#### 選択肢4：破棄する

**先に確認する：**
```
以下が完全に削除されます：
- ブランチ <名前>
- すべてのコミット：<コミットリスト>
- <パス> のワークツリー

確認するには 'discard' と入力してください。
```

正確な確認を待つ。

確認された場合：
```bash
MAIN_ROOT=$(git -C "$(git rev-parse --git-common-dir)/.." rev-parse --show-toplevel)
cd "$MAIN_ROOT"
```

その後：ワークツリーをクリーンアップ（ステップ6）、そしてブランチを強制削除：
```bash
git branch -D <フィーチャーブランチ>
```

### ステップ6：ワークスペースをクリーンアップする

**選択肢1と4のみで実行する。** 選択肢2と3は常にワークツリーを保持する。

```bash
GIT_DIR=$(cd "$(git rev-parse --git-dir)" 2>/dev/null && pwd -P)
GIT_COMMON=$(cd "$(git rev-parse --git-common-dir)" 2>/dev/null && pwd -P)
WORKTREE_PATH=$(git rev-parse --show-toplevel)
```

**`GIT_DIR == GIT_COMMON` の場合：** 通常リポジトリ。クリーンアップするワークツリーなし。完了。

**ワークツリーのパスが `.worktrees/`・`worktrees/`・`~/.config/superpowers/worktrees/` 配下の場合：** Superpowers がこのワークツリーを作成した — クリーンアップの責任はこちらにある。

```bash
MAIN_ROOT=$(git -C "$(git rev-parse --git-common-dir)/.." rev-parse --show-toplevel)
cd "$MAIN_ROOT"
git worktree remove "$WORKTREE_PATH"
git worktree prune  # 自己修復：古い登録をクリーンアップ
```

**それ以外の場合：** ホスト環境（ハーネス）がこのワークスペースを管理している。削除してはいけない。プラットフォームにワークスペース終了ツールがある場合はそれを使用する。なければそのままにする。

## クイックリファレンス

| 選択肢 | マージ | プッシュ | ワークツリー保持 | ブランチ削除 |
|--------|--------|----------|----------------|------------|
| 1. ローカルマージ | yes | - | - | yes |
| 2. PR作成 | - | yes | yes | - |
| 3. そのままにする | - | - | yes | - |
| 4. 破棄 | - | - | - | yes（強制） |

## よくある間違い

**テスト確認をスキップする**
- **問題：** 壊れたコードをマージ、失敗するPRを作成
- **修正：** 選択肢を提示する前に必ずテストを確認する

**オープンエンドな質問をする**
- **問題：** 「次に何をすべきですか？」は曖昧
- **修正：** 正確に4つの選択肢を提示する（デタッチHEADの場合は3つ）

**選択肢2でワークツリーをクリーンアップする**
- **問題：** PRの反映のためにユーザーが必要とするワークツリーを削除する
- **修正：** 選択肢1と4のみクリーンアップする

**ワークツリーを削除する前にブランチを削除する**
- **問題：** ワークツリーがまだブランチを参照しているため `git branch -d` が失敗する
- **修正：** まずマージ、ワークツリーを削除、その後ブランチを削除

**ワークツリーの中から git worktree remove を実行する**
- **問題：** カレントディレクトリが削除対象のワークツリーの内部にあるとコマンドが失敗する
- **修正：** `git worktree remove` の前に必ずメインリポジトリのルートに `cd` する

**ハーネスが作成したワークツリーをクリーンアップする**
- **問題：** ハーネスが作成したワークツリーを削除すると幽霊状態が発生する
- **修正：** `.worktrees/`・`worktrees/`・`~/.config/superpowers/worktrees/` 配下のワークツリーのみクリーンアップする

**破棄の確認をしない**
- **問題：** 作業を誤って削除する
- **修正：** 「discard」の入力確認を要求する

## 禁止事項

**絶対にしないこと：**
- テストが失敗した状態で進む
- マージ結果のテスト確認なしにマージする
- 確認なしに作業を削除する
- 明示的なリクエストなしに強制プッシュする
- マージ成功を確認する前にワークツリーを削除する
- 自分が作成していないワークツリーをクリーンアップする（出所確認）
- ワークツリーの内部から `git worktree remove` を実行する

**必ずすること：**
- 選択肢を提示する前にテストを確認する
- メニューを提示する前に環境を検出する
- 正確に4つの選択肢を提示する（デタッチHEADの場合は3つ）
- 選択肢4には入力確認を得る
- 選択肢1と4のみワークツリーをクリーンアップする
- ワークツリー削除前にメインリポジトリのルートに `cd` する
- 削除後に `git worktree prune` を実行する

# コードレビューの受け取り

## 概要

コードレビューは技術的な評価であり、感情的なパフォーマンスではない。

**コア原則：** 実装する前に確認する。仮定する前に尋ねる。社会的快適さより技術的正確さを優先する。

## レスポンスパターン

```
コードレビューのフィードバックを受け取ったとき：

1. 読む：反応せずにフィードバックを完全に読む
2. 理解する：要件を自分の言葉で言い直す（または確認する）
3. 確認する：コードベースの実態と照合する
4. 評価する：このコードベースに技術的に適切か？
5. 応答する：技術的な確認または理由のある反論
6. 実装する：一項目ずつ、各項目をテストする
```

## 禁止レスポンス

**絶対に言ってはいけない：**
- 「おっしゃる通りです！」（明示的な CLAUDE.md 違反）
- 「良い指摘ですね！」「素晴らしいフィードバックです！」（パフォーマンス的）
- 「今すぐ実装します」（確認前）

**代わりに：**
- 技術的な要件を言い直す
- 明確化の質問をする
- 間違っている場合は技術的な理由で反論する
- ただ作業を始める（行動 > 言葉）

## 不明確なフィードバックへの対応

```
いずれかの項目が不明確な場合：
  停止 — まだ何も実装しない
  不明確な項目について明確化を求める

理由：項目が関連している可能性がある。部分的な理解 = 間違った実装。
```

**例：**
```
ヒューマンパートナー：「1〜6を修正して」
1・2・3・6は理解できる。4・5は不明確。

❌ 間違い：1・2・3・6を今実装し、4・5は後で尋ねる
✅ 正解：「1・2・3・6は理解しました。進める前に4と5について明確化が必要です。」
```

## ソース別の対応

### ヒューマンパートナーからの場合
- **信頼する** — 理解した後に実装する
- **それでも尋ねる** — スコープが不明確な場合
- **パフォーマンス的な同意はしない**
- **行動か技術的な確認にスキップする**

### 外部レビュアーからの場合
```
実装前に：
  1. 確認：このコードベースに技術的に正確か？
  2. 確認：既存の機能を壊すか？
  3. 確認：現在の実装の理由は何か？
  4. 確認：すべてのプラットフォーム/バージョンで動くか？
  5. 確認：レビュアーは完全なコンテキストを理解しているか？

提案が間違っていると思う場合：
  技術的な理由で反論する

容易に確認できない場合：
  そう言う：「[X] なしではこれを確認できません。[調査/尋ねる/進める] のどれにしますか？」

ヒューマンパートナーの以前の決定と競合する場合：
  まずヒューマンパートナーと相談する
```

**ヒューマンパートナーのルール：** 「外部フィードバック — 懐疑的に、でも慎重に確認する」

## 「プロ仕様」機能への YAGNI チェック

```
レビュアーが「適切に実装する」を提案した場合：
  コードベースで実際の使用状況を grep する

  使用されていない場合：「このエンドポイントは呼ばれていません。削除しますか（YAGNI）？」
  使用されている場合：適切に実装する
```

**ヒューマンパートナーのルール：** 「あなたもレビュアーも私に報告する。この機能が必要ないなら追加しない。」

## 実装順序

```
複数項目のフィードバックの場合：
  1. まず不明確なものをすべて明確化する
  2. その後この順番で実装する：
     - ブロッキング問題（壊れるもの、セキュリティ）
     - 簡単な修正（タイポ、インポート）
     - 複雑な修正（リファクタリング、ロジック）
  3. 各修正を個別にテストする
  4. リグレッションがないことを確認する
```

## 反論すべきとき

以下の場合に反論する：
- 提案が既存の機能を壊す
- レビュアーが完全なコンテキストを持っていない
- YAGNI に違反する（未使用の機能）
- このスタックに技術的に不正確
- レガシー/互換性の理由がある
- ヒューマンパートナーのアーキテクチャ決定と競合する

**反論の方法：**
- 防衛的にならず技術的な理由を使う
- 具体的な質問をする
- 機能するテスト/コードを参照する
- アーキテクチャに関わる場合はヒューマンパートナーを関与させる

**反論を声に出しにくい場合のシグナル：** 「Circle K で不思議なことが起きている」

## 正しいフィードバックへの対応

フィードバックが正しい場合：
```
✅ 「修正しました。[何が変わったかの簡単な説明]」
✅ 「良い発見です — [具体的な問題]。[場所] で修正しました。」
✅ [ただコードで修正を見せる]

❌ 「おっしゃる通りです！」
❌ 「良い指摘ですね！」
❌ 「見つけてくれてありがとうございます！」
❌ 「[何かについて] ありがとうございます」
❌ 感謝の表現はすべて禁止
```

**感謝しない理由：** 行動が語る。ただ修正する。コード自体がフィードバックを聞いたことを示す。

**「ありがとう」と書きそうになったら：** 削除する。代わりに修正内容を述べる。

## 反論を優雅に訂正する

反論したが間違っていた場合：
```
✅ 「あなたが正しかった — [X] を確認して [Y] であることが分かりました。今実装します。」
✅ 「これを確認して正しいです。最初の理解が間違っていた理由は [理由] です。修正します。」

❌ 長い謝罪
❌ なぜ反論したかの弁護
❌ 過度な説明
```

事実として訂正を述べて次へ進む。

## よくある間違い

| 間違い | 修正 |
|--------|------|
| パフォーマンス的な同意 | 要件を述べるかただ行動する |
| 盲目的な実装 | まずコードベースと照合する |
| テストなしでバッチ処理 | 一度に一つ、各項目をテストする |
| レビュアーが正しいと仮定する | 物事を壊すかどうか確認する |
| 反論を避ける | 技術的正確さ > 快適さ |
| 部分的な実装 | まずすべての項目を明確化する |
| 確認できないのに進む | 制限を述べて方向性を聞く |

## 実例

**パフォーマンス的な同意（悪い例）：**
```
レビュアー：「レガシーコードを削除して」
❌ 「おっしゃる通りです！削除します...」
```

**技術的な確認（良い例）：**
```
レビュアー：「レガシーコードを削除して」
✅ 「確認中...ビルドターゲットは10.15以上、このAPIには13以上が必要。後方互換性のためにレガシーが必要。バンドルIDが間違っている — 修正するか13以前のサポートを切りますか？」
```

**YAGNI（良い例）：**
```
レビュアー：「データベース・日付フィルター・CSVエクスポートを使った適切なメトリクス追跡を実装して」
✅ 「コードベースを grep した — このエンドポイントを呼ぶものが何もない。削除しますか（YAGNI）？使用箇所が漏れていますか？」
```

**不明確な項目（良い例）：**
```
ヒューマンパートナー：「1〜6を修正して」
1・2・3・6は理解できる。4・5は不明確。
✅ 「1・2・3・6は理解しました。実装前に4と5について明確化が必要です。」
```

## GitHubスレッドの返信

GitHub のインラインレビューコメントに返信する場合、トップレベルのPRコメントとしてではなく、コメントスレッドで返信する（`gh api repos/{owner}/{repo}/pulls/{pr}/comments/{id}/replies`）。

## まとめ

**外部フィードバック = 評価すべき提案であり、従うべき命令ではない。**

確認する。疑問を持つ。そして実装する。

パフォーマンス的な同意はしない。常に技術的な厳密さを持つ。

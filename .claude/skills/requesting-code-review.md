# コードレビューのリクエスト

問題が連鎖する前に捕捉するためにコードレビューのサブエージェントをディスパッチする。レビュアーは評価のために正確に作られたコンテキストを得る — 自分のセッション履歴は渡さない。これによりレビュアーは思考プロセスではなく作業成果物に集中でき、自分のコンテキストも継続作業のために保持できる。

**コア原則：** 早めにレビューし、頻繁にレビューする。

## レビューをリクエストするとき

**必須：**
- サブエージェント駆動の開発における各タスクの後
- 主要機能の完了後
- main へのマージ前

**任意だが価値がある：**
- 行き詰まったとき（新鮮な視点）
- リファクタリング前（ベースラインチェック）
- 複雑なバグ修正後

## リクエスト方法

**1. git SHA を取得する：**
```bash
BASE_SHA=$(git rev-parse HEAD~1)  # または origin/main
HEAD_SHA=$(git rev-parse HEAD)
```

**2. コードレビューのサブエージェントをディスパッチする：**

`general-purpose` タイプの Task ツールを使用し、`code-reviewer.md` のテンプレートを記入する

**プレースホルダー：**
- `{DESCRIPTION}` — 作ったものの簡単なサマリー
- `{PLAN_OR_REQUIREMENTS}` — 何をすべきだったか
- `{BASE_SHA}` — 開始コミット
- `{HEAD_SHA}` — 終了コミット

**3. フィードバックに対応する：**
- Critical の問題は即座に修正する
- Important の問題は進む前に修正する
- Minor の問題は後で対応するようメモする
- レビュアーが間違っている場合は反論する（理由を添えて）

## 例

```
[タスク2完了：検証関数の追加]

自分：コードレビューをリクエストしてから進みます。

BASE_SHA=$(git log --oneline | grep "Task 1" | head -1 | awk '{print $1}')
HEAD_SHA=$(git rev-parse HEAD)

[コードレビューサブエージェントをディスパッチ]
  DESCRIPTION：4つの問題タイプを持つ verifyIndex() と repairIndex() を追加
  PLAN_OR_REQUIREMENTS：docs/superpowers/plans/deployment-plan.md のタスク2
  BASE_SHA：a7981ec
  HEAD_SHA：3df7661

[サブエージェントが返答]：
  強み：クリーンなアーキテクチャ、実際のテスト
  問題：
    Important：進捗インジケーターが欠けている
    Minor：レポート間隔のマジックナンバー（100）
  評価：進めて問題なし

自分：[進捗インジケーターを修正]
[タスク3へ続く]
```

## ワークフローとの連携

**サブエージェント駆動の開発：**
- 各タスクの後にレビューする
- 問題が重なる前に捕捉する
- 次のタスクへ進む前に修正する

**計画の実行：**
- 各タスクまたは自然なチェックポイントの後にレビューする
- フィードバックを受け取り、適用し、続ける

**アドホックな開発：**
- マージ前にレビューする
- 行き詰まったときにレビューする

## 禁止事項

**絶対にしないこと：**
- 「単純だから」レビューをスキップする
- Critical の問題を無視する
- 未修正の Important の問題を抱えて進む
- 有効な技術的フィードバックに反論する

**レビュアーが間違っている場合：**
- 技術的な理由で反論する
- 機能することを証明するコード/テストを見せる
- 明確化を求める

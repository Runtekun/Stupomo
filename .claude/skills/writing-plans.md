# 実装計画の作成

## 概要

エンジニアがコードベースのコンテキストをまったく持たず、趣味も疑わしいと仮定して、包括的な実装計画を書く。各タスクに触れるべきファイル・コード・テスト・確認すべきドキュメント・テスト方法を含めたすべての計画を文書化する。実現可能な小さなタスクの計画全体を提供する。DRY。YAGNI。TDD。頻繁なコミット。

ツールセットや問題ドメインについてほとんど何も知らない熟練した開発者を想定する。良いテスト設計はあまり得意でないと仮定する。

**開始時に宣言する：** 「実装計画を作成するために writing-plans スキルを使用します。」

**コンテキスト：** 分離されたワークツリーで作業している場合、実行時に `superpowers:using-git-worktrees` スキルで作成されているべきだ。

**計画の保存先：** `docs/superpowers/plans/YYYY-MM-DD-<機能名>.md`
- （計画の場所に関するユーザーの好みがこのデフォルトをオーバーライドする）

## スコープの確認

仕様が複数の独立したサブシステムをカバーする場合、ブレインストーミング中にサブプロジェクトの仕様に分割されているべきだった。されていない場合、別々の計画に分割することを提案する — サブシステムごとに1つ。各計画は独自に動作するテスト可能なソフトウェアを生成すべきだ。

## ファイル構造

タスクを定義する前に、作成または変更されるファイルとその責任をマップアウトする。ここで分解の決定が確定する。

- 明確な境界と明確なインターフェースを持つ単位を設計する。各ファイルには1つの明確な責任があるべきだ。
- 一度にコンテキストに持てるコードについて最も良く推論でき、ファイルが集中しているときに編集はより信頼性が高い。大きすぎて何でもやるファイルよりも小さく集中したファイルを好む。
- 一緒に変わるファイルは一緒にあるべきだ。技術的なレイヤーではなく責任で分割する。
- 既存のコードベースでは確立されたパターンに従う。コードベースが大きなファイルを使っているなら勝手に再構成しない。ただし変更するファイルが扱いにくくなっている場合、計画に分割を含めることは合理的だ。

この構造がタスクの分解を知らせる。各タスクは独立して意味をなす自己完結した変更を生成すべきだ。

## 小さなタスクの粒度

**各ステップは1つのアクション（2〜5分）：**
- 「失敗するテストを書く」 - ステップ
- 「失敗することを確認するために実行する」 - ステップ
- 「テストを通過させる最小限のコードを実装する」 - ステップ
- 「テストを実行して通過することを確認する」 - ステップ
- 「コミット」 - ステップ

## 計画ドキュメントのヘッダー

**すべての計画はこのヘッダーで始まらなければならない：**

```markdown
# [機能名] 実装計画

> **エージェントワーカーへ：** 必須サブスキル：superpowers:subagent-driven-development（推奨）または superpowers:executing-plans を使用してこの計画をタスクごとに実装する。ステップはチェックボックス（`- [ ]`）構文で追跡する。

**ゴール：** [これが何を構築するかの一文の説明]

**アーキテクチャ：** [アプローチについての2〜3文]

**技術スタック：** [主要な技術/ライブラリ]

---
```

## タスク構造

````markdown
### タスクN：[コンポーネント名]

**ファイル：**
- 作成：`exact/path/to/file.py`
- 変更：`exact/path/to/existing.py:123-145`
- テスト：`tests/exact/path/to/test.py`

- [ ] **ステップ1：失敗するテストを書く**

```python
def test_specific_behavior():
    result = function(input)
    assert result == expected
```

- [ ] **ステップ2：失敗することを確認するためにテストを実行する**

実行：`pytest tests/path/test.py::test_name -v`
期待：FAIL「function not defined」

- [ ] **ステップ3：最小限の実装を書く**

```python
def function(input):
    return expected
```

- [ ] **ステップ4：通過することを確認するためにテストを実行する**

実行：`pytest tests/path/test.py::test_name -v`
期待：PASS

- [ ] **ステップ5：コミット**

```bash
git add tests/path/test.py src/path/file.py
git commit -m "feat: specific feature を追加"
```
````

## プレースホルダーなし

すべてのステップには、エンジニアが必要とする実際のコンテンツが含まれていなければならない。これらは**計画の失敗**だ — 絶対に書いてはいけない：
- 「TBD」「TODO」「後で実装」「詳細を記入」
- 「適切なエラーハンドリングを追加する」/「バリデーションを追加する」/「エッジケースを処理する」
- 「上記のテストを書く」（実際のテストコードなし）
- 「タスクNと同様」（コードを繰り返す — エンジニアはタスクを順不同で読むかもしれない）
- コードステップでどうするかを説明するが方法を示さないステップ（コードのステップにはコードブロックが必要）
- どのタスクでも定義されていない型・関数・メソッドへの参照

## 覚えておくこと
- 常に正確なファイルパス
- すべてのステップに完全なコード — ステップがコードを変更する場合は、コードを示す
- 期待される出力を含む正確なコマンド
- DRY、YAGNI、TDD、頻繁なコミット

## セルフレビュー

完全な計画を書いた後、新鮮な目で仕様を見て、それに対して計画を確認する。これは自分で実行するチェックリストだ — サブエージェントのディスパッチではない。

**1. 仕様のカバレッジ：** 仕様の各セクション/要件をざっと見る。それを実装するタスクを指差せるか？ギャップをリストアップする。

**2. プレースホルダーのスキャン：** 計画を「プレースホルダーなし」セクションの赤信号パターンで検索する。修正する。

**3. 型の一貫性：** 後のタスクで使用した型・メソッドシグネチャ・プロパティ名は、前のタスクで定義したものと一致しているか？ タスク3では `clearLayers()` だがタスク7では `clearFullLayers()` というのはバグだ。

問題が見つかったらその場で修正する。再レビューは不要。修正して次へ進む。仕様の要件にタスクがない場合、タスクを追加する。

## 実行の引き渡し

計画を保存した後、実行の選択肢を提供する：

**「計画は完成して `docs/superpowers/plans/<ファイル名>.md` に保存されました。2つの実行オプションがあります：**

**1. サブエージェント駆動（推奨）** — タスクごとに新鮮なサブエージェントをディスパッチし、タスク間でレビュー、高速イテレーション

**2. インライン実行** — executing-plans を使用してこのセッションでタスクを実行し、チェックポイントでのバッチ実行

**どちらのアプローチにしますか？」**

**サブエージェント駆動を選んだ場合：**
- **必須サブスキル：** superpowers:subagent-driven-development を使用する
- タスクごとに新鮮なサブエージェント + 2段階レビュー

**インライン実行を選んだ場合：**
- **必須サブスキル：** superpowers:executing-plans を使用する
- チェックポイントでのバッチ実行

# 体系的なデバッグ

## 概要

ランダムな修正は時間を無駄にし、新しいバグを生み出す。素早いパッチは根本的な問題を隠す。

**コア原則：** 修正を試みる前に常に根本原因を見つける。症状の修正は失敗だ。

**このプロセスの文言に違反することは、デバッグの精神に違反することだ。**

## 鉄則

```
根本原因の調査なしに修正を提案してはならない
```

フェーズ1を完了していない場合、修正を提案することはできない。

## 使用するとき

以下のあらゆる技術的問題に使用する：
- テスト失敗
- 本番環境のバグ
- 予期しない動作
- パフォーマンスの問題
- ビルド失敗
- 統合の問題

**特に以下の場合に使用する：**
- 時間的プレッシャー下（緊急事態は推測を誘惑する）
- 「ちょっとした簡単な修正」が明白に見える
- すでに複数の修正を試みた
- 以前の修正が機能しなかった
- 問題を完全に理解していない

**スキップしてはいけない場合：**
- 問題が単純に見える（単純なバグにも根本原因がある）
- 急いでいる（急ぐことは手戻りを保証する）
- マネージャーが今すぐ直したいと言っている（体系的な方法はやり直しより速い）

## 4つのフェーズ

次のフェーズに進む前に各フェーズを完了しなければならない。

### フェーズ1：根本原因の調査

**修正を試みる前に：**

1. **エラーメッセージを注意深く読む**
   - エラーや警告を読み飛ばさない
   - 正確な解決策が含まれていることが多い
   - スタックトレースを完全に読む
   - 行番号・ファイルパス・エラーコードをメモする

2. **一貫して再現する**
   - 確実に引き起こせるか？
   - 正確な手順は何か？
   - 毎回起きるか？
   - 再現できない場合 → データをさらに収集する、推測しない

3. **最近の変更を確認する**
   - これを引き起こす可能性のある変更は何か？
   - git diff、最近のコミット
   - 新しい依存関係、設定変更
   - 環境の違い

4. **複数コンポーネントシステムでエビデンスを収集する**

   **システムが複数のコンポーネントを持つ場合（CI → ビルド → サイニング、API → サービス → データベース）：**

   **修正を提案する前に、診断用の計装を追加する：**
   ```
   各コンポーネント境界について：
     - コンポーネントに入るデータをログに記録する
     - コンポーネントから出るデータをログに記録する
     - 環境/設定の伝播を確認する
     - 各レイヤーの状態を確認する

   どこで壊れるかを示すエビデンスを収集するために一度実行する
   その後、エビデンスを分析して失敗しているコンポーネントを特定する
   その後、その特定のコンポーネントを調査する
   ```

   **例（多層システム）：**
   ```bash
   # レイヤー1：ワークフロー
   echo "=== ワークフローで利用可能なシークレット ==="
   echo "IDENTITY: ${IDENTITY:+SET}${IDENTITY:-UNSET}"

   # レイヤー2：ビルドスクリプト
   echo "=== ビルドスクリプトの環境変数 ==="
   env | grep IDENTITY || echo "IDENTITYが環境にない"

   # レイヤー3：サイニングスクリプト
   echo "=== キーチェーン状態 ==="
   security list-keychains
   security find-identity -v

   # レイヤー4：実際のサイニング
   codesign --sign "$IDENTITY" --verbose=4 "$APP"
   ```

   **これにより判明する：** どのレイヤーが失敗するか（シークレット → ワークフロー ✓、ワークフロー → ビルド ✗）

5. **データフローを追跡する**

   **エラーがコールスタックの深い部分にある場合：**

   完全な後退追跡テクニックは、このディレクトリの `root-cause-tracing.md` を参照。

   **簡単なバージョン：**
   - 悪い値はどこに由来するか？
   - 何がその悪い値でこれを呼んだか？
   - ソースを見つけるまで追跡し続ける
   - 症状ではなくソースを修正する

### フェーズ2：パターン分析

**修正する前にパターンを見つける：**

1. **機能している例を見つける**
   - 同じコードベースで類似した機能するコードを探す
   - 壊れているものと類似した機能するものは何か？

2. **参照と比較する**
   - パターンを実装している場合、参照実装を完全に読む
   - 流し読みしない — 一行一行読む
   - 適用する前にパターンを完全に理解する

3. **違いを特定する**
   - 機能するものと壊れているものの違いは何か？
   - どんなに小さくても、すべての違いをリストアップする
   - 「それは関係ない」と仮定しない

4. **依存関係を理解する**
   - これはどのような他のコンポーネントを必要とするか？
   - どんな設定・コンフィグ・環境？
   - どんな仮定をしているか？

### フェーズ3：仮説とテスト

**科学的方法：**

1. **単一の仮説を立てる**
   - 明確に述べる：「X が根本原因だと思う。なぜなら Y だから」
   - 書き留める
   - 具体的に、曖昧にしない

2. **最小限にテストする**
   - 仮説をテストするための最小限の変更をする
   - 一度に一つの変数
   - 複数のものを同時に修正しない

3. **続ける前に確認する**
   - 機能したか？ はい → フェーズ4
   - 機能しなかったか？ 新しい仮説を立てる
   - さらに修正を積み重ねない

4. **分からない場合**
   - 「X が分からない」と言う
   - 知っているふりをしない
   - ヘルプを求める
   - もっと調べる

### フェーズ4：実装

**症状ではなく根本原因を修正する：**

1. **失敗するテストケースを作成する**
   - できる限り最小限の再現
   - 可能であれば自動テスト
   - フレームワークがない場合は一回限りのテストスクリプト
   - 修正前に必ず用意する
   - 適切な失敗テストを書くには `superpowers:test-driven-development` スキルを使用する

2. **単一の修正を実装する**
   - 特定された根本原因に対処する
   - 一度に一つの変更
   - 「ついでに」改善しない
   - バンドルされたリファクタリングなし

3. **修正を確認する**
   - テストが今度は通過するか？
   - 他のテストが壊れていないか？
   - 問題が実際に解決されたか？

4. **修正が機能しない場合**
   - 停止する
   - 数える：いくつの修正を試みたか？
   - 3件未満の場合：フェーズ1に戻り、新しい情報で再分析する
   - **3件以上の場合：停止してアーキテクチャに疑問を持つ（以下のステップ5）**
   - アーキテクチャの議論なしに4件目の修正を試みない

5. **3件以上の修正が失敗した場合：アーキテクチャに疑問を持つ**

   **アーキテクチャの問題を示すパターン：**
   - 各修正が別の場所で新しい共有状態/結合/問題を明らかにする
   - 修正には「大規模なリファクタリング」が必要
   - 各修正が別の場所に新しい症状を作り出す

   **停止して根本を問い直す：**
   - このパターンは根本的に健全か？
   - 「惰性でそれを続けている」のか？
   - 症状を修正し続けるべきかアーキテクチャをリファクタリングすべきか？

   **さらに修正を試みる前にヒューマンパートナーと議論する**

   これは失敗した仮説ではない — これは間違ったアーキテクチャだ。

## 要注意サイン — 停止してプロセスに従う

以下を考えていたら：
- 「今は簡単に直して、後で調査する」
- 「Xを変えてみてうまくいくか見てみよう」
- 「複数の変更を加えて、テストを実行する」
- 「テストをスキップして、手動で確認する」
- 「おそらくXだろう、それを修正しよう」
- 「完全には理解していないが、これで動くかもしれない」
- 「パターンはXと言っているが、違う方法で適用する」
- 「主な問題は：[調査なしに修正をリストアップする]」
- データフローを追跡する前に解決策を提案している
- **「もう一度修正を試みる」（すでに2件以上試みた場合）**
- **各修正が別の場所で新しい問題を明らかにする**

**これらすべては：停止する。フェーズ1に戻る。**

**3件以上の修正が失敗した場合：** アーキテクチャに疑問を持つ（フェーズ4.5参照）

## ヒューマンパートナーの「間違っている」シグナル

**以下のリダイレクションに注意する：**
- 「それは起きていないの？」 — 確認せずに仮定した
- 「それは見せてくれる...？」 — エビデンス収集を追加すべきだった
- 「推測するのをやめて」 — 理解せずに修正を提案している
- 「これをウルトラシンクして」 — 症状だけでなく根本を疑問視する
- 「詰まってる？」（フラストレーション）— アプローチが機能していない

**これらを見たら：** 停止する。フェーズ1に戻る。

## よくある合理化

| 言い訳 | 現実 |
|--------|------|
| 「問題は単純、プロセスは不要」 | 単純な問題にも根本原因がある。単純なバグにはプロセスが速い。 |
| 「緊急事態、プロセスの時間がない」 | 体系的なデバッグは当てずっぽうより速い。 |
| 「まずこれを試して、それから調査する」 | 最初の修正がパターンを作る。最初から正しくやる。 |
| 「修正が機能することを確認してからテストを書く」 | テストなしの修正は定着しない。最初にテストが証明する。 |
| 「同時に複数の修正で時間を節約する」 | 何が機能したか分離できない。新しいバグを引き起こす。 |
| 「参照が長すぎる、パターンを適用する」 | 部分的な理解はバグを保証する。完全に読む。 |
| 「問題が見えた、修正しよう」 | 症状を見ることは根本原因を理解することではない。 |
| 「もう一度修正を試みる」（2件以上失敗後） | 3件以上の失敗 = アーキテクチャの問題。修正を続けず、パターンを疑問視する。 |

## クイックリファレンス

| フェーズ | 主な活動 | 成功基準 |
|--------|---------|---------|
| **1. 根本原因** | エラーを読む・再現する・変更を確認する・エビデンスを収集する | 何が・なぜ起きているかを理解する |
| **2. パターン** | 機能する例を見つける・比較する | 違いを特定する |
| **3. 仮説** | 理論を立てる・最小限にテストする | 確認または新しい仮説 |
| **4. 実装** | テストを作成する・修正する・確認する | バグ解決、テスト通過 |

## プロセスで「根本原因なし」が判明した場合

体系的な調査で問題が本当に環境的・タイミング依存・外部的であると判明した場合：

1. プロセスを完了した
2. 調査したことを文書化する
3. 適切な処理を実装する（リトライ・タイムアウト・エラーメッセージ）
4. 将来の調査のためにモニタリング/ロギングを追加する

**しかし：** 「根本原因なし」のケースの95%は調査が不完全だ。

## サポートテクニック

これらのテクニックは体系的なデバッグの一部であり、このディレクトリで利用可能：

- **`root-cause-tracing.md`** — コールスタックを後退してバグの元のトリガーを見つける
- **`defense-in-depth.md`** — 根本原因を見つけた後、複数のレイヤーで検証を追加する
- **`condition-based-waiting.md`** — 任意のタイムアウトを条件ポーリングに置き換える

**関連スキル：**
- **superpowers:test-driven-development** — 失敗するテストケースを作成する（フェーズ4、ステップ1）
- **superpowers:verification-before-completion** — 修正が機能したことを主張する前に確認する

## 現実の効果

デバッグセッションより：
- 体系的なアプローチ：修正まで15〜30分
- ランダムな修正アプローチ：2〜3時間の試行錯誤
- 一発修正率：95%対40%
- 新しいバグの導入：ゼロ近く対一般的
